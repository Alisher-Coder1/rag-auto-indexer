Автоматическое перестроение FAISS-индекса при изменении knowledge_base.txt (MVP)
Основание: финальное ТЗ (разделы 5–7) ⏱ 8–12 часов (1–2 дня)
✅ Завершённый этап / веха 🔄 В работе ⏳ Запланировано ⚠️ Риск / точка внимания
🔍 Этап 1 – Подготовка и анализ
⏱ 1–2 часа 🎯 Устранить неопределённости
Цель: определить реальные пути, имена файлов индекса и сигнатуры существующих функций.
Задача Детали Время Критерий готовности
1.1 Найти и зафиксировать пути Определить директорию индекса, имена файлов 20 мин Константы INDEX_DIR, KNOWLEDGE_BASE_PATH, HASH_STORE_PATH

🔐 Этап 2 – Hash-механизм и вспомогательные функции
⏱ 2–3 часа 🎯 Безопасные утилиты хеширования/JSON
Создать масштабируемые, отказоустойчивые функции: чтение чанками, атомарная запись, очистка индекса.
Функция Особенности Время Критерий
compute_file_hash SHA256, чанки 8192 байт 20 мин Работает с файлами любого размера
load_kb_hash Чтение JSON, обработка JSONDecodeError + вывод [WARN] 20 мин Битый JSON → None + предупреждение
save_kb_hash Атомарная запись через .tmp + os.replace 20 мин При сбое не остаётся битого файла
faiss_index_exists Проверка существования всех файлов индекса 10 мин True только при наличии обоих файлов
_clean_index_dir Удаление всех файлов в INDEX_DIR, создание пустой директории 20 мин Директория пуста и существует
Модульные тесты (опционально) Проверка функций изоляции 1 ч Все функции работают корректно
✅ Веха 2: Хеш-механизм готов – можно интегрировать в get_db_index.
⚙️ Этап 3 – Модификация get_db_index()
⏱ 3–4 часа 🎯 Основная логика + защита загрузки
Реализовать принятие решений, rebuild с очисткой, защиту загрузки индекса (try/except + один повторный rebuild).
Задача Детали Время Критерий готовности
3.1 Проверка KB os.path.exists, иначе FileNotFoundError 10 мин Отсутствие файла → исключение
3.2 Логика need_rebuild Проверка существования индекса, хеша, сравнение 30 мин Корректно 3 сценария
3.3 Информационные сообщения На английском (INFO, WARN, ERROR) 15 мин Появляются в нужные моменты
3.4 Rebuild с очисткой Вызов _clean_index_dir перед build_and_save_faiss_index 20 мин Полная перезапись индекса
3.5 Обработка ошибок rebuild try/except, вывод [ERROR], выброс RuntimeError 20 мин При падении хеш не сохраняется
3.6 Защита загрузки индекса try/except для load_faiss_index, при ошибке → один повторный rebuild 1 ч Битый индекс автоматически перестраивается (один раз)
3.7 Интеграция Собрать полную версию get_db_index 30 мин Функция выполняет всю логику
✅ Веха 3: get_db_index() полностью готова к тестированию.
🧪 Этап 4 – Ручное тестирование (по ТЗ)
⏱ 2–3 часа 🎯 Проверка 10+ сценариев
Убедиться, что система работает корректно во всех граничных случаях.
1. Первый запуск → сборка + JSON
2. Повторный запуск → загрузка, без сборки
3. Изменение KB → перестроение
4. Нет kb_hash.json, индекс есть → rebuild + WARN
5. Битый kb_hash.json → rebuild + WARN
6. Нет индекса, хеш есть → перестроение
7. Повреждён индекс (удалён один файл) → перестроение
8. Ошибка в build_and_save → исключение, старый индекс не меняется
9. Ошибка загрузки индекса → авто rebuild → загрузка
10. Нет knowledge_base.txt → FileNotFoundError
✅ Веха 4: Все тест-кейсы пройдены (включая работу с большими файлами).
📦 Этап 5 – Подготовка к сдаче
⏱ 1 час 🎯 Финальный код, документация
Задача Детали Время Критерий готовности
5.1 Добавить docstring Все новые/изменённые функции 20 мин Каждая функция описана
5.2 Удалить отладочный код Закомментированные print, временные логи 10 мин Чистый код
5.3 Проверить константы Относительные пути, вынесены в начало модуля 5 мин Нет абсолютных путей
5.4 Финальный прогон Тест-кейсы 1,2,3 в чистом окружении 20 мин Всё работает с нуля
5.5 README_changes.md (опционально) Краткое описание поведения 5 мин Документ для проверяющего
✅ Веха 5: Код готов к сдаче.
