# Техническое задание (финальная версия)

## Автоматическое перестроение FAISS-индекса при изменении knowledge_base.txt

---

## 1. Цель

Реализовать механизм автоматического обнаружения изменений в файле `knowledge_base.txt` и, при наличии изменений или отсутствии индекса, пересоздавать FAISS-индекс без участия разработчика. Механизм должен работать при каждом обращении к индексу через функцию `get_db_index()`.

**Критерий успеха**:

- При изменении `knowledge_base.txt` следующий вызов `get_db_index()` автоматически перестраивает индекс.
- При отсутствии изменений индекс загружается мгновенно (без перестроения).
- В консоль выводятся информационные сообщения на **английском языке** о всех значимых действиях и ошибках.

---

## 2. Границы задачи

### Входит в текущее Pro-задание (MVP)

- Хеширование файла `knowledge_base.txt` алгоритмом **SHA256** (с чтением чанками по 8 КБ).
- Хранение хеша в JSON-файле `kb_hash.json` (корень проекта) с атомарной записью через временный файл.
- Проверка актуальности индекса при каждом вызове `get_db_index()`.
- Автоматический rebuild при изменении хеша, отсутствии индекса или отсутствии/повреждении `kb_hash.json`.
- Вывод консольных сообщений о статусе (только на английском).
- Базовая обработка ошибок: битый JSON, падение `build_and_save_faiss_index`, битый индекс при загрузке.
- Перед rebuild – полная очистка директории индекса для предотвращения частичных состояний.
- При ошибке загрузки индекса – автоматическая попытка перестроения (один раз).

### НЕ входит в текущее задание (отложено)

- Асинхронная/фоновая перестройка индекса.
- Версионирование, снапшоты, откаты.
- Валидация целостности индекса (контрольные суммы).
- Health-check эндпоинты.
- Graceful recovery с сохранением старого индекса.
- Поддержка нескольких документов.

---

## 3. Обязательные требования задания

1. **Хеширование** – SHA256, чтение файла чанками (8192 байт).
2. **Хранение хеша** – `kb_hash.json` с полями `hash` и `last_updated` (ISO8601). Запись атомарная: сначала во временный файл `kb_hash.json.tmp`, затем `os.replace`.
3. **Проверка при каждом вызове** – `get_db_index()` перед загрузкой индекса сравнивает хеши.
4. **Автоматический rebuild при изменении хеша** – вызывает `build_and_save_faiss_index`, затем обновляет JSON.
5. **Автоматический build при отсутствии индекса** – если индексной директории или требуемых файлов не существует.
6. **Информационные сообщения в консоль (английский)**:
   - `[INFO] Index not found, rebuilding...`
   - `[INFO] knowledge_base.txt changed, rebuilding index...`
   - `[INFO] Index is up-to-date, loading existing.`
   - `[WARN] Failed to read kb_hash.json, will rebuild.`
   - `[ERROR] Failed to rebuild index: <details>`
   - `[ERROR] Failed to load index: <details>`
   - `[INFO] Attempting to rebuild index due to load failure...`

- `docstore.json` – сериализованный DocumentStore

## 5. Этапы внедрения

### Этап 1 – Подготовка

- Определить точные имена файлов индекса в существующем коде.
- Зафиксировать константы путей.
- Убедиться, что `build_and_save_faiss_index` создаёт директорию при необходимости.
- Выделить загрузку индекса в функцию `load_faiss_index`, если её нет.

### Этап 2 – Реализация хеш-механизма

- Написать `compute_file_hash` (с чанками).
- Написать `load_kb_hash` (с обработкой `JSONDecodeError` и выводом WARN).
- Написать `save_kb_hash` (атомарная запись).
- Написать `faiss_index_exists` (проверка двух файлов).
- Написать `_clean_index_dir` (очистка директории индекса).

### Этап 3 – Модификация `get_db_index`

Реализовать логику:

1. Проверить существование `knowledge_base.txt` – иначе `FileNotFoundError`.
2. Вычислить `current_hash`, загрузить `saved_hash`.
3. Определить `need_rebuild` по условиям (отсутствие индекса, отсутствие хеша, несовпадение хешей).
4. Если `need_rebuild`:
   - Вызвать `_clean_index_dir(INDEX_DIR)`
   - Вызвать `build_and_save_faiss_index`
   - Вызвать `save_kb_hash`
   - При любой ошибке – вывести `[ERROR]` и выбросить исключение.
5. Загрузка индекса с защитой:
   - `try: return load_faiss_index(INDEX_DIR)`
   - `except Exception as e:` вывести `[ERROR] Failed to load index`, затем выполнить один повторный rebuild (очистка + сборка + сохранение хеша) и повторно загрузить. Если повторная ошибка – выбросить исключение.

### Этап 4 – Тестирование

Прогнать тест-кейсы из раздела 7.

### Этап 5 – Подготовка к сдаче

- Добавить docstring.
- Удалить отладочные print.
- Проверить, что все сообщения на английском.

## 6. Definition of Done (DoD)

- [ ] Код реализован согласно этапам 2 и 3.
- [ ] При первом запуске создаются индекс и `kb_hash.json` с корректным хешем, вывод `[INFO] Index not found, rebuilding...`.
- [ ] При повторном запуске без изменений – загрузка без rebuild, вывод `[INFO] Index is up-to-date, loading existing.`

- [ ] При ошибке в `build_and_save_faiss_index` – выбрасывается исключение, старый индекс не повреждается.
- [ ] При ошибке загрузки индекса – автоматическая попытка перестроения (один раз), при успехе – загрузка нового, при повторной ошибке – исключение.
- [ ] Все файлы читаются/пишутся безопасно: хеш – чанками, JSON – атомарно, индекс – с очисткой перед сборкой.
